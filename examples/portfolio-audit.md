## After (projected: 82/100 with these fixes)

### Fix 1 — Add tests to `api-project` (+4 per repo)

Even a basic test suite signals engineering maturity.
For a Python/FastAPI project, 5 integration tests covering happy path + error cases is enough.

```bash
# Minimal test file to get started
tests/
  test_api.py   # tests for your main endpoints
```

### Fix 2 — Write a "Why I built this" section in each README (+4 per repo)

Three sentences minimum:
1. What problem you personally encountered
2. Why existing solutions weren't good enough
3. What's different because this exists

Bad: "This is a REST API for managing user data."
Good: "I built this to solve a recurring problem I noticed while interning at [X]: no lightweight tool existed for [specific thing]. Existing solutions required [expensive or complex thing]. This API does [specific thing] in under [time/lines/setup steps]."

### Fix 3 — Add architecture section to `api-project` (+4)

Even a text diagram is enough:

```
Client → FastAPI → PostgreSQL
              ↓
         Redis Cache (TTL: 5 min)
```

One paragraph on why you made this design choice.

### Fix 4 — Deploy `cli-tool` and add install instructions (+4)

A pip-installable CLI with a one-line install is a strong demo:

```bash
pip install your-tool
your-tool --help
```

Or publish to PyPI and add the badge.

### Fix 5 — Add one metric to `api-project` (+4)

Options:
- "Handles 500 req/s in load test on a t3.micro"
- "94% test coverage"
- "Reduced [something] from X to Y"

If you don't have real numbers, run a quick benchmark and report it honestly.
